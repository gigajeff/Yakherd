## P1 Findings

### P1-1: Build and test code retains PyPI OIDC authority

The single `publish` job grants `id-token: write` at job scope
(`.github/workflows/publish.yml:12-18`). That authority is therefore available
while the same job checks out candidate source, installs build dependencies,
runs repository and package tests, runs acceptance, executes the build backend,
and smoke-installs both archives (`publish.yml:19-62`). Only after all of that
does the job invoke the publisher (`publish.yml:63-66`).

This gives candidate scripts and transitive build dependencies the ability to
request a GitHub OIDC token under the same repository, workflow, tag, and
environment identity trusted by PyPI. The dedicated `pypi` environment and
exact release-tag binding constrain which job invocation may publish, but they
do not isolate the publishing credential from earlier code in that invocation.
A compromised build or test input could mint and exchange the token before the
reviewed publisher step.

This is not merely a stylistic preference. The official PyPA action guidance
for v1.14.0 says to grant `id-token: write` only to the publishing job and to
separate building from publishing so malicious build/test scripts cannot
elevate privileges. It describes distributions as normally prepared in an
earlier restricted job. A PyPA maintainer separately characterizes giving
transitive build dependencies OIDC impersonation power as dangerous and
unsupported. GitHub's current Python publishing example likewise uses a
separate `release-build` job and an OIDC-enabled `pypi-publish` job:

- https://github.com/pypa/gh-action-pypi-publish/tree/v1.14.0#trusted-publishing
- https://github.com/pypa/gh-action-pypi-publish/tree/v1.14.0#non-goals
- https://github.com/pypa/gh-action-pypi-publish/discussions/349
- https://docs.github.com/en/actions/tutorials/build-and-test-code/python#publishing-to-pypi

Required fix: put checkout, validation, dependency installation, build,
distribution inspection, and exact-archive smoke tests in a job without
`id-token: write`; transfer only the already-tested `dist/` files through
SHA-pinned upload/download artifact actions; and give a dependent `pypi`
environment job only the permissions and steps needed to retrieve and publish
those exact files. Preserve the existing tag/version/commit binding and ensure
the artifact handoff remains exact and fail closed. Then rerun the complete
release evidence and request a fresh independent review.

## Failed 1.1.0 Workflow And Publisher-Pin Assessment

The GitHub Actions job and decoded log for run
https://github.com/gigajeff/Yakherd/actions/runs/29782686682 independently
confirm the stated failure mode:

- tag, release, tests, acceptance, build, distribution inspection, wheel smoke,
  and sdist smoke steps all completed successfully;
- the sole failing step was `Publish package distributions to PyPI`;
- the action set `REF` to
  `6733eb7d741f0b11ec6a39b58540dab7590f9b7d`, generated a Docker action using
  that ref, and attempted to run
  `ghcr.io/pypa/gh-action-pypi-publish:6733eb7d741f0b11ec6a39b58540dab7590f9b7d`;
- Docker failed before container startup with `manifest unknown`.

Because the publisher container never started, no publisher code, Trusted
Publishing exchange, attestation, or distribution upload occurred. The 1.1.1
changelog accurately records this failed/no-upload attempt.

The replacement pin
`cef221092ed1bacb1cc03d23a2d87d1d172e277b` is the immutable official source
commit for v1.14.0 and has a matching published GHCR image tag. At that commit,
`action.yml` supplies `github.action_ref` as `REF`, and
`create-docker-action.py` maps it to `docker://ghcr.io/{repo}:{REF}`. The new
pin therefore resolves the existing immutable source/image pair while retaining
SHA pinning:

- https://github.com/pypa/gh-action-pypi-publish/tree/cef221092ed1bacb1cc03d23a2d87d1d172e277b
- https://github.com/pypa/gh-action-pypi-publish/blob/cef221092ed1bacb1cc03d23a2d87d1d172e277b/action.yml
- https://github.com/pypa/gh-action-pypi-publish/blob/cef221092ed1bacb1cc03d23a2d87d1d172e277b/create-docker-action.py
- https://github.com/pypa/gh-action-pypi-publish/pkgs/container/gh-action-pypi-publish/789435266?tag=v1.14.0

The pin repair is correct for the observed container-resolution failure, but it
does not close P1-1.

## Release Verdict

Verdict: `fail` for the exact current Yakherd 1.1.1 PyPI recovery candidate.
The immutable v1.14.0 source/image pin correctly addresses the observed
`manifest unknown` failure, version and hash bindings are internally correct,
and the completed local package/archive gates pass. P1-1 nevertheless leaves
PyPI OIDC authority exposed to build, test, and candidate code, violating the
project's fail-closed release priorities and the publisher's security guidance.

The exact current 1.1.1 candidate must not be committed as release-ready or
tagged, even if ordinary remote CI passes. Remote CI cannot cure this
source-level privilege boundary. Correct P1-1, rerun all required tests,
acceptance, positive/negative release verification, exact archive verification,
wheel and sdist smoke installs, and `git diff --check`, then obtain a fresh
independent Red Team review before commit, tag, or publication.

human action required: split unprivileged build/test/archive production from the OIDC-enabled publish job, preserve exact artifact binding with SHA-pinned handoff actions, regenerate and rerun the complete evidence, and request a clean independent recheck.
