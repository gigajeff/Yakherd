## P1 Findings

### P1-1: The Trusted Publishing job is not bound to a reviewed release tag and publishes artifacts it never smoke-tests

The release policy requires the wheel and source distribution to be built and
smoke-installed in clean environments, followed by an intentional release
commit and `vMAJOR.MINOR.PATCH` tag before PyPI publication
(`docs/RELEASE.md:12-16`). The proposed publish job instead accepts a manual
dispatch, requires only that the dispatch ref be `refs/heads/main`, and never
checks that `GITHUB_SHA` is the commit named by `v1.1.0` or by any version-matched
tag (`.github/workflows/publish.yml:3-14`). Thus a later main-tip commit can be
published under the authority of this workflow without being the exact commit
that an independent review or release tag authenticated.

The job runs source distribution/package tests and acceptance before installing
the build frontend and before creating the wheel and sdist. It then sends the
new outputs directly to PyPI without installing, invoking, inspecting, or
hashing either output (`.github/workflows/publish.yml:24-39`). The job also
resolves an unpinned `build` installation at publish time and uses a moving
`pypa/gh-action-pypi-publish@release/v1` reference. A protected `pypi`
environment and OIDC limit who can publish, but they do not establish that the
published archive bytes are the tagged, smoke-tested candidate bytes.

Impact: this breaks the exact-candidate authentication boundary at the final
publication step. Passing source tests cannot prove the behavior or contents of
artifacts built afterward, and branch protection alone cannot substitute for a
version/tag/commit binding. This is release-blocking.

Required fix: make the publish path fail closed on a version-matched release
tag and verify that the tag resolves to the executing commit. Build once, inspect
the archive contents, smoke-install the exact wheel and exact sdist in clean
environments, exercise the installed command, and publish only those same
artifacts. Pin the build and publishing supply-chain inputs to reviewed versions
or immutable digests. Re-review the corrected workflow and exact release
candidate before publication.

## Cold-Resume Changes

The two governed cold-resume changes are technically sound within their stated
scope.

- The pure protocol validator no longer reads the ambient wall clock. It parses
  `STATUS.md` and evidence timestamps structurally and requires UTC without
  computing age (`validate_protocol.py:303-317,434-445`). Source inspection
  found no remaining `datetime.now`, `date.today`, or equivalent ambient-clock
  call in that validator. The new negative fixture covers an old but valid UTC
  timestamp and rejection of a naive timestamp.
- The bootstrap review prompt now authorizes five deterministic JSON run-record
  names, requires the shared run-record schema, supplies each of the first four
  records as a separate `--evidence` argument, records the evidence check, and
  separately authorizes one review path. This closes the prior conflict between
  a review-only writer and the repository's structured-evidence completion
  contract without authorizing product or authority-owner edits.

The manifest pins all four changed template files. Current
`MANIFEST.json` SHA-256 is
`9713244b63ca1c6518f914bef97ec805a027110c0d1fc24fbad7ed297356587c`,
which is exactly bound by `RELEASE.json`; `bootstrap.py` remains
`c41f4f5d11e5eb3bfed9eabfdef7444e28612d124a26101df8bfb3628837a8bb`.
