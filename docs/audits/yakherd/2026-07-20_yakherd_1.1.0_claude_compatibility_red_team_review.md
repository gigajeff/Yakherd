## P2 Findings

### P2-1: The absolute-beginner Claude path omits two conditions required for the imported rules to load

The adapter itself is correct, but the public beginner instructions describe
its effect as unconditional. `README.md:49` says Claude Code automatically
loads `CLAUDE.md`, which imports `AGENTS.md`; `README.md:54-56` says the adapters
make the rules load automatically; and the Claude beginner prompt at
`README.md:69-75` asks only for confirmation that the generated import exists.
The same unconditional wording appears in `docs/USAGE.md:12-18`,
`packages/jeff_strict_ssot_v1/README.md:23-31`, and the installed
`template/README.md`.

Claude Code's current official documentation confirms that `@AGENTS.md` is the
recommended Windows-compatible bridge, but it also states that imported files
are loaded at launch, that the first external import encountered in a project
causes an approval dialog, and that declining leaves imports disabled. It also
states that project `CLAUDE.md` discovery is based on the working-directory
hierarchy. See [How Claude remembers your project](https://code.claude.com/docs/en/memory),
especially “Import additional files,” “AGENTS.md,” and “How CLAUDE.md files
load.” Therefore, a beginner who remains in the installer session/repository,
or declines the first import prompt without understanding it, can have a valid
installed `CLAUDE.md` while the authoritative `AGENTS.md` is not present in the
active Claude context. The validator can prove the adapter's bytes, but cannot
prove this client-side approval/session state.

Required fix: qualify Claude support as native once the generated repository is
opened as the Claude Code project and the local `AGENTS.md` import is approved.
The copy/paste Claude prompt should direct the agent to start a fresh session in
the generated repository, explain and safely handle the import approval, and
verify the loaded instruction sources with Claude Code's documented `/context`
or `/memory` view before the bootstrap Red Team task. Equivalent session-root
wording should be added wherever automatic loading is claimed. Because the
installed `template/README.md` is governed package source, regenerate
`MANIFEST.json`, `RELEASE.json`, clean acceptance evidence, and independent
review after the correction.

## P3 Findings

### P3-1: The prerequisite wording can be read as saying Codex and Claude Code bundle Git and Python

`README.md:39-42` correctly lists filesystem, shell, Git, and Python 3.11+
requirements, then says Codex and Claude Code “provide those capabilities.”
The agents provide file/shell integration, but availability of Git and Python
at the required version belongs to the local or hosted execution environment;
it is not guaranteed merely by choosing either client. For an absolute
beginner, change this to “Codex or Claude Code can do this when their execution
environment also has Git and Python 3.11+,” and let the installation prompt ask
the agent to verify prerequisites before cloning or writing.

## Adapter And Single-Authority Assessment

The proposed adapter design passes. Root `CLAUDE.md`, source-template
`template/CLAUDE.md`, and acceptance-installed `CLAUDE.md` are each exactly 11
bytes, contain only `@AGENTS.md` plus LF, and share SHA-256
`336cc4fbf19beaada7ccf9986414fa91851a8d7a07dfb3ccbe800a69eed0ab49`.
Anthropic's official documentation explicitly recommends this adapter for a
repository whose authority already lives in `AGENTS.md`, including on Windows.
No duplicate Claude-specific rules or second authority were introduced.

The generated validator makes `CLAUDE.md` a required exact-case,
non-reparse, in-repository regular file and rejects content other than the
`@AGENTS.md` import (`validate_protocol.py:16-56,139-150,388-413`). Its negative
fixture rejects both a missing adapter and duplicated/divergent rules
(`test_validate_protocol.py:113-123`). The package installation test verifies
the exact rendered adapter and the 1.1.0 receipt identity
(`tests/test_bootstrap.py:94-102`). These are appropriate repository-side
controls; the remaining approval/session condition is necessarily documented
client behavior rather than an installer-enforceable security boundary.

## README Compatibility Assessment

The revised README otherwise makes the requested distinctions accurately:
Yakherd supplies the governance/reliability half, the user's idea or master
prompt supplies the product half, an ordinary chat without repository and
shell tools cannot install it, generic agents require an explicit instruction
to read `AGENTS.md`, and Yakherd does not guarantee generated product
correctness. The trust-boundary correction also accurately limits the retained
Mosaic-origin audits to the 1.0.0 bytes.

Codex's native claim is supported by [OpenAI's AGENTS.md documentation](https://developers.openai.com/codex/guides/agents-md),
which states that Codex reads repository `AGENTS.md` before work. Claude's
adapter claim is structurally supported by Anthropic's documentation. P2-1 is
limited to the missing first-use and session-root conditions in the beginner
workflow, not a rejection of first-class Claude Code compatibility.

## Verdict And Release Gate

Verdict: `fail` for release readiness of the exact current 1.1.0 candidate.

The package bytes, adapter, manifest/release chain, installed tree, validators,
and bounded acceptance evidence pass. The candidate is not yet suitable for
the promised absolute-beginner Claude workflow because its public instructions
omit a documented state in which the import is disabled or not loaded. Correct
P2-1 (and preferably P3-1), regenerate the governed hashes and clean acceptance
evidence, and obtain an independent re-review of those exact bytes before the
release commit. A documentation-only assertion cannot waive the package cycle
when `template/README.md` changes.

## Residual Risks

- `CLAUDE.md` and `AGENTS.md` are behavioral context, not an enforcement
  sandbox. Claude user, local, ancestor, managed, or path-scoped instructions
  can coexist and conflict; Codex can likewise receive higher-scope guidance.
- Coding-agent clients and their instruction-discovery rules can change. Public
  compatibility claims should remain qualified and periodically checked
  against official vendor documentation.
- The review proves repository bytes and bounded installer/validator behavior,
  not strict model compliance or the correctness and safety of software later
  generated from a master prompt.
- The current working tree also contains untracked `yakherd2.png`, which was
  outside this compatibility candidate and was not reviewed. Explicit staging
  must exclude it unless it receives separate review and authorization.
