### Prior P1-1: closed

The prior review found that publication was manually dispatched from `main`,
was not bound to the reviewed release tag/commit, and published artifacts that
the workflow had not inspected or smoke-installed. The current
`.github/workflows/publish.yml` closes each part of that boundary:

- only a pushed `v*.*.*` tag triggers the job; there is no manual-dispatch
  trigger (`publish.yml:3-7`);
- the job requires a tag ref, checks checked-out `HEAD` against `GITHUB_SHA`,
  resolves the named tag to the same commit, and passes that tag to the release
  verifier (`publish.yml:23-28`);
- `scripts/verify_release.py --tag` requires the exact
  `v<pyproject-version>` value, while the same verifier binds PyPI version,
  package version, bootstrap, manifest, and template hashes;
- the build frontend and its direct dependencies are exact-version pinned, the
  setuptools build requirement is exact-version pinned, and checkout,
  Python-setup, and PyPI-publish actions use immutable 40-character commit SHAs;
- one `python -m build` invocation creates the release archives; the next step
  verifies those exact files against all 49 reviewed package files;
- separate clean environments install the exact wheel and exact sdist from
  `dist/`, execute their installed `yakherd` commands, and strictly validate
  both generated repositories before the final publisher step; and
- the publisher consumes the same `dist/` outputs through the dedicated `pypi`
  environment with only `id-token: write` added to the job.

Positive and negative local checks confirmed that `v1.1.0` passes the release
verifier and `v1.1.1` fails with an exact tag mismatch. The tag workflow has not
been run during this review; its eventual remote success remains a release
gate, not evidence retroactively inferred from local inspection.
