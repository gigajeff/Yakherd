## Findings

### P0

No P0 finding was identified.

### P1-1: A process can escape before Job assignment and before durable identity recording

`_run_in_job` creates the child suspended at
`src/yakherd/process_hygiene.py:855`, but does not durably write the child's
identity until after Job assignment at line 886. If the broker is terminated
between `Popen` and assignment, the on-disk `starting` record has no child
identity, the empty Job handle closes, and startup reconciliation has no PID to
verify or terminate. The suspended, unassigned child can remain indefinitely.

The ordinary error path has the same gap when `process_identity(process.pid)`
returns `None`: line 863 raises, but the fallback at lines 939-957 terminates
the child only when `record["top_process"]` is a dictionary. The existing
assignment-failure test injects failure later, after identity acquisition, so
it does not cover either escape. This violates the accepted assignment-failure,
abnormal-exit, recoverability, and zero-owned-descendant requirements.

### P1-2: Verified cleanup has a PID-reuse time-of-check/time-of-use race

`_terminate_verified_process` verifies identity through `process_identity` at
lines 552-557. That helper opens and closes one process handle. The termination
helper then opens a second handle by PID at lines 558-565 and calls
`TerminateProcess` without revalidating creation time and executable identity
on that same handle. If the verified process exits and Windows reuses its PID
between those operations, cleanup can terminate the unrelated replacement.

The stale-PID test covers a mismatch that already exists before cleanup; it
does not cover reuse between verification and termination. The accepted safety
invariant requires that unrelated processes are never killed, so verification
and termination must be bound to the same opened process identity.

### P1-3: The finite-command boundary accepts non-finite timeouts and known wrapped persistent forms

`run_broker` rejects only timeouts `<= 0` at lines 775-776. Python floating
point `nan` and positive `inf` both pass that check; `argparse` accepts those
spellings. An infinite run timeout never expires, and a NaN deadline also never
satisfies `now >= deadline`. Queue timeout has the same defect. Therefore the
CLI does not require a finite timeout as the accepted Definition of Done
requires.

The command filter is also only a shallow top-level token check. For example,
`powershell -Command "npm run dev"` is accepted, and `cmd /c "start tool.exe"`
is accepted when the command tail is supplied as one normal argument. These
are exact prohibited forms in the accepted contract and installed policy. The
tests cover only their direct tokenizations. Together with non-finite timeout
acceptance, this provides a direct fail-open path to the persistent work the
policy claims to reject.

### P1-4: The Stop/SubagentStop hook does not clean a stopped task whose broker remains alive

The hook validates only `hook_event_name`, discards the supplied session,
turn, and working-directory scope, and calls `status_report` at lines
1290-1294. Reconciliation explicitly skips every record whose broker identity
is still alive at lines 1024-1026. Such a task is merely reported active; it is
not a cleanup failure, so the hook returns `continue: true` and leaves its Job
running.

The accepted contract requires deterministic cleanup on agent stop and
subagent stop, with a current-turn-scoped cleanup entry point that does not
affect unrelated work. The candidate neither binds task ownership to the hook
payload nor terminates a verified current-turn Job. Its hook test exercises
only a completed task and an already-stale ambiguous record, not an active
stopped task.

### P1-5: The supplied Windows tests do not satisfy the accepted completion condition

The accepted contract says not to report completion until Windows acceptance
tests demonstrate baseline recovery after repeated builds, cancellations,
timeouts, nested trees, and abnormal exits, and its stress case calls for
hundreds of mixed short commands, cancelled builds, and nested trees. The
candidate's stress test runs 40 successful one-process commands. There is no
direct interruption/cancellation test, no mixed repeated timeout/nested/crash
stress sequence, and the abnormal-exit test waits until the record is already
`running`, so it cannot expose the pre-assignment crash window in P1-1.

The reported 17 focused-test PASS is valid for the cases implemented, but it
does not prove the explicit completion gate or the candidate documentation's
every-exit-path guarantee.

### Cycle-1 finding dispositions

#### P1-1: RESOLVED — atomic containment removes the pre-assignment escape

The corrected broker no longer uses the create-suspended/assign/resume
sequence. `_create_process_in_job` supplies the new per-task Job handle through
`PROC_THREAD_ATTRIBUTE_JOB_LIST` to `CreateProcessW`, so Windows assigns the
child before its first instruction. The function verifies membership on the
returned process handle and, on any post-create failure, terminates and waits
on that same handle before closing it. A broker crash after creation closes the
kill-on-close Job rather than leaving an unassigned child. Tests verify that
the child's first instruction observes Job membership and that an injected
membership-verification failure leaves verified-zero cleanup.

#### P1-2: RESOLVED — verification and termination use one process handle

`_terminate_verified_process` now opens one handle with query, synchronize, and
terminate rights, derives PID/creation-time/executable identity from that exact
handle, compares it with the record, and calls `TerminateProcess` only on the
same handle. PID reuse after the handle is opened cannot redirect termination
to a replacement process. The stale-identity test continues to prove mismatch
fail-closed behavior.

#### P1-3: RESOLVED — timeout and recognized wrapped-command paths fail closed

`run_broker` now applies `math.isfinite` to both execution and queue timeouts
before accepting positive values; NaN and either infinity are rejected. The
finite-command filter now examines PowerShell, cmd, bash/sh, and WSL wrapper
payloads for the accepted known watcher/server/background forms, including the
two cycle-1 examples. Tests exercise non-finite values and direct and wrapped
prohibited commands. The installed policy accurately limits this lexical
guardrail: opaque scripts are not claimed to be semantically proven finite;
the mandatory finite timeout and Job remain the lifecycle backstop.

#### P1-4: RESOLVED — Stop/SubagentStop performs owner-scoped active cleanup

When `CODEX_THREAD_ID` is available, new records bind owner to
`codex:<thread-id>`. The hook requires non-empty event, session, turn, and
working-directory fields; selects only records owned by the emitting session;
requests cancellation; terminates the exact task Job and same-handle-verified
members; and verifies no Job or recorded process remains. Active same-workspace
tasks with a different owner are left untouched and make the hook fail closed.
Windows tests cover successful active-session cleanup and different-owner
preservation/failure.

#### P1-5: RESOLVED — the accepted lifecycle completion evidence is present

The Windows suite now directly tests Ctrl-Break cancellation with exit 130 and
verified-zero Job cleanup. Its 200-run mixed lifecycle test includes normal
finite commands, nested child trees, timeouts, cancellation markers, and broker
crashes, then verifies every record clean, zero active/queued/failure status,
and the broker handle count within the defined baseline tolerance. Separate
tests cover atomic first-instruction containment, concurrent heavy queuing,
parent-before-child, unrelated Node preservation, and the session hook paths.
This closes the cycle-1 coverage gap for the accepted MVP.
