## Goal

Correct Y-PROC-1's identity and cleanup-severity model without weakening its
Windows Job Object containment. PID reuse, missing legacy identity, and
internally contradictory records must become non-actionable warnings instead
of termination targets or global execution embargoes. Only a coherent,
currently verified, task-owned process whose cleanup failed and whose concrete
hazard is independently established may block unrelated work.

## Root-cause correction

Core 1.3 correctly made the Job Object the runtime containment boundary and
used PID plus creation time and path before direct termination. Its
reconciliation layer nevertheless treated identity mismatch as an ambiguous
global failure, retained completed records in the active registry, and had no
formal distinction between diagnostic uncertainty and a verified hazardous
cleanup failure. The correction introduces coherent process snapshots, a
formal state/severity classifier, prompt retirement from active tracking, and
an explicit warning-acknowledgement/resume operation.

## Architecture decision

- The unique per-task Windows Job Object remains the primary ownership
  boundary. Process records support reconciliation and same-handle fallback;
  ancestry, image names, command-line similarity, and bare PIDs never create
  ownership.
- Active records use schema `yakherd.process-task.v1.1`; the loader accepts
  legacy `v1` records for non-actionable classification and retirement.
- A coherent observation records PID, creation time, executable path, derived
  image name, command line when safely available, parent PID and parent
  creation time when available, task ID, execution/session ID, exact Job name
  and membership, observation time, and lifecycle.
- Reconciliation classifies each observation as `VERIFIED_LIVE_OWNED`,
  `VERIFIED_EXITED`, `PID_REUSED_UNRELATED`, `OWNERSHIP_UNVERIFIED`, or
  `OWNERSHIP_RECORD_INCONSISTENT`. PID reuse and uncertainty are warnings and
  are removed from the actionable active set.
- Cleanup severity is `CLEANUP_WARNING` unless coherent live Job ownership,
  failed cleanup, and independently recorded concrete hazard evidence all
  agree. Only that conjunction is `CLEANUP_BLOCKER`.
- Terminal records move from `tasks/` to `history/`. History remains
  diagnostic evidence but is never an active PID target.
- `yakherd process resume` acknowledges stable warning fingerprints once and
  permits progress. It refuses a blocker with its concrete hazard explanation.
- Persistent leases remain unsupported in this amendment; the state model
  reserves `APPROVED_PERSISTENT` without creating lease behavior.

## Definition of Done

1. The amendment is preserved in `docs/` and linked as the corrective owner
   from the original package and current public policy documentation.
2. Every new process observation contains the coherent identity fields above;
   records with contradictory or missing ownership evidence are classified
   without termination or execution embargo.
3. PID absence, PID reuse, path mismatch, legacy missing identity, parent PID
   reuse, Job membership, and post-termination reuse follow the amendment's
   formal state transitions.
4. Active tracking retires terminal and stale records promptly. History PIDs
   cannot become cleanup targets.
5. Status, cleanup, reconciliation, and hooks expose warnings separately from
   blockers; only a verified live owned cleanup failure with concrete hazard
   evidence can block new work.
6. Explicit resume acknowledges warnings without repeat authorization and
   refuses a verified concrete blocker with its reason.
7. Windows tests cover all fifteen amendment cases, including live process
   creation, cancellation, conflict metadata, rapid reuse simulation, CPU
   hazard, harmless survival, and resume behavior.
8. Core/package version is `1.3.1`; hashes and fresh acceptance evidence are
   regenerated; required process, package, distribution, release, and
   whitespace checks pass.
9. Independent Red Team review of the exact candidate reports no P0/P1.
10. Final status has zero active verified hazardous descendants and zero
    cleanup blockers, and Git/downstream boundaries are reported honestly.

## Explicit limitation

Y-PROC-1.1 can prove process identity and Job membership. Concrete non-CPU
resource hazards require explicit, process-bound evidence in the task record;
this amendment does not guess file, port, device, or database ownership from
names or ancestry. Absent that proof, the anomaly is a warning rather than a
global blocker.
