## Goal

Add the smallest standard-library Windows execution governor that makes
`Y-PROC-1` usable from the Yakherd CLI and inherited by fresh Yakherd-governed
repositories through a compact policy reference.

## Architecture decision

The public `yakherd` package owns execution governance; the audited installer
continues to render files only and never runs product commands. `yakherd exec`
uses `CreateProcessW` with `PROC_THREAD_ATTRIBUTE_JOB_LIST` to assign the child
to a per-task kill-on-close Job Object atomically at creation, before its first
instruction. Heavy jobs are created at below-normal priority. Per-task JSON
records live in user-local Yakherd state outside governed repositories. A
user-scoped lock serializes heavy top-level pipelines while preserving their
internal parallelism.

The installed template adds only:

- a short `AGENTS.md` invariant requiring the broker; and
- `.yakherd/policies/Y-PROC-1.md`, which owns detailed usage, hook guidance,
  migration, and deferred scope.

Yakherd will expose a hook cleanup entry point, but the installer will not
write Codex hook configuration. Creating automation remains outside the
installer trust boundary and requires a separate, explicit opt-in. Codex
project hooks are trusted-project configuration and non-managed hooks require
user review, so silently installing them would neither be enforceable nor
consistent with Yakherd's guarantees.

## Authorized write boundary

- focused distribution/process tests under `tests/`;

- package manifest/release bindings and tests required by those template
  changes; and
- one exact-diff Red Team review record under `docs/audits/yakherd/`.

## Forbidden scope

- no changes to `mosaic_colmap`, `SPLATOMATIC`, or `CROCHET`;
- no SSOT refresh or replacement of their temporary protections;
- no network, dependency installation, product execution, automation, or
  target-repository Git behavior in the installer;
- no broad process-name termination;
- no fixed two-worker cap or modification of internal build parallelism;
- no adaptive CPU budgeting, general scheduler, telemetry history, UI, or
  rich persistent-process leases in this MVP; and
- no release, publication, staging, commit, push, tag, or remote mutation.

## Definition of Done

1. `yakherd exec` rejects known interactive/detached/watch/server forms,
   requires a finite timeout, and runs finite commands in a Windows Job Object.
2. Heavy commands are one-at-a-time across the current Windows user, run at
   `BELOW_NORMAL_PRIORITY_CLASS`, and retain normal child parallelism.
3. Success, failure, timeout, interrupt, and broker exit close or terminate the
   owned job and verify zero job members; unrelated processes are never
   selected by name.
4. Registry identity includes PID, creation time, executable, command, working
   directory, owner, lifecycle, job identity, exit state, and cleanup result.
5. `yakherd process status`, dry-run cleanup, verified owned cleanup, startup
   reconciliation, and a session-scoped Stop/SubagentStop-compatible hook
   handler exist.
6. Windows tests cover atomic first-instruction containment, nested children,
   simultaneous heavy requests, finite and non-finite timeouts, direct
   cancellation, parent-before-child, wrapped forbidden commands, stale PID
   identity, unrelated processes, broker abnormal exit, scoped hook cleanup,
   and 200 mixed lifecycle/handle-stress runs.
7. Fresh generated repositories contain the compact policy reference and
   policy owner, while existing repositories are documented as preserve-first
   reviewed retrofits.
8. Package hashes and acceptance evidence are regenerated; all required checks
   pass; one independent Red Team review of the exact diff has no P0/P1.

## Explicit MVP limitation

Approved persistent processes remain unsupported and fail closed. A future
lease feature must be separately scoped and reviewed; Y-PROC-1 v1.3 permits
only finite attached commands. Codex hooks are defense-in-depth because some
specialized tool paths can bypass the normal hook path; Job Object cleanup in
the broker remains the primary lifecycle boundary.
